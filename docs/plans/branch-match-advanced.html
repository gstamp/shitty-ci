Advanced branch matching for .shitty-ci.yml

Executive summary: Today, branches: is a single allow-list of path-style globs with OR semantics. That is great for simple cases, but awkward for common policies like build every branch except protected lines. This plan expands the design space on purpose: wildcards remain first-class, and a small, bounded mini language is an acceptable implementation path if it improves clarity over YAML-only boolean plumbing.

1. Clear problem definition

Scan this first

  • Who: Repo owners using shitty-ci on a workstation/home server.
  • Pain: allow-only lists force you to say what you want in the positive, which explodes for "everything except …".
  • Why it matters: mis-stated rules waste CPU, spam statuses, or silently skip the one branch you cared about.
  • Success signal: a human can predict the outcome from reading 3–8 lines of config, and logs explain the decision.

The poller already reads .shitty-ci.yml from the candidate commit and calls ShouldBuildRef before enqueueing work. The gap is expressiveness in config—not scheduling, workspaces, or GitHub posting.

2. Goals and non-goals

Goals

  • Express "build all branches except X" without enumerating the world.
  • Keep wildcard / glob matching as a supported primitive (including richer wildcards if we extend beyond path.Match—see open questions).
  • Allow a bounded mini expression language if it reduces YAML noise (still parser-friendly, testable, and documented).
  • Preserve backward compatibility for today's list-only branches: configs.
  • Optional parity for tags: if we change branch matching holistically.

Non-goals

  • A Turing-complete policy DSL, arbitrary code hooks, or cross-field logic ("if branch X and file Y changed").
  • Org-wide policy inheritance, templating marketplace, or external policy servers.
  • UI/schema editors, migration wizards, or GitHub App changes.
  • Supporting non-origin remotes in this iteration (unless already tracked elsewhere).

Explicitly not ruled out anymore

Regex can be acceptable if bounded (length limits, compile timeouts, clear errors). The earlier "regex is always a non-goal" stance was too strong for this project's scale.

3. User-centered scenarios

  • Scenario A — feature branch hygiene: "Run CI on every branch except main."
  • Scenario B — deploy branches: "Build everything except staging and production on this host."
  • Scenario C — release lines: "Match release/* but skip release/experimental-*."
Example session expand
  1. User edits .shitty-ci.yml, pushes.
  2. Poller evaluates feature/foo → queue.
  3. Push to main → skip log names the rule that fired (expression vs pattern—TBD).

4. Precise functional requirements

Represent each rule as a Go table test in internal/shittyyml (plus parser tests if we add an expression string).

  • FR-1: With no branch configuration, behavior matches today: build all remote branches.
  • FR-2: Any new syntax must define an unambiguous precedence story (include vs exclude vs expression).
  • FR-3: If multiple representations are accepted (list + expression), they must either be mutually exclusive in YAML or have deterministic composition rules—documented, not tribal.
  • FR-4: Invalid rules fail closed: no build enqueue, actionable daemon log (as today for invalid YAML / glob errors).
  • FR-5: Normalization stays aligned with today: strip refs/remotes/origin/, ignore …/HEAD.
  • FR-6: README gets a few vivid examples (not a manual).

6. Success metrics

  • M-1: Every semantic branch is covered by tests (including failure cases).
  • M-2: "All except main/staging/prod" fits in a small config block without hacks.
  • M-3: Skip logs remain understandable without reading Go source.

7. Constraints and tradeoffs

  • Human readability beats cleverness: a mini language is fine; an obscure puzzle language is not.
  • Hot path cost: evaluation should remain roughly O(number of rules) per ref check.
  • Safety: if we add regex or richer patterns, enforce limits (pattern length, compile time, no catastrophic backtracking engines).
  • YAML foot-guns: prefer errors over silently ignoring conflicting keys.

8. Technical context (not overdesign)

Today: internal/shittyyml → File.ShouldBuildRef → matchAnyGlob using path.Match. Tomorrow: keep that core, but add a composable layer (exclude lists and/or a parsed expression AST).

flowchart TD A[Poll tip SHA for refs/remotes/origin/*] --> B{refs/tags/* ?} B -->|yes| T[Existing tag rules] B -->|no| C{remote branch?} C -->|no| S[Do not build] C -->|yes| D[Normalize branch name] D --> E{Invalid rule / parse error?} E -->|yes| X[Return error; skip and log reason] E -->|no| F{Rule mode} F -->|legacy list| L[OR glob match on branches list] F -->|include/exclude| I[include set then exclude set] F -->|mini expression| M[Evaluate bounded AST] L --> Q[Queue / skip] I --> Q M --> Q
Diagram note

Final chart should reflect the one composition rule we ship. The point of this revision: expression mode is allowed, but must remain a narrow, testable evaluator—not a second product.

9. YAML design options (progressive disclosure)

These are intentionally colorful: pick primary vs alternates during implementation.

Primary candidate Option A — explicit include / exclude maps

Great for scanning; wildcards live naturally in both buckets.

branches:
  include:
    - "*"
  exclude:
    - main
    - staging
    - production

Alternate Option B — parallel list with negation markers

Very compact. If branch names can start with !, require quoting or escaping rules.

branches:
  - "*"
  - "!main"
  - "!staging"

Power-user Option C — bounded mini expression

One string (or small AST in YAML) that names operations like all, except, any, and leaves wildcards inside operands.

branches_when: 'all() except ["main", "staging", "production"]'

Syntax above is illustrative only: ship something tokenizable in under a millisecond for typical strings, with golden tests for precedence and quoting.

Option D — separate top-level exclude key (compat-friendly) 
branches:
  - "*"
branches_exclude:
  - main
  - staging

Keeps branches: as a list for older templates; pairs naturally with wildcards.

Guardrail (still true)

Avoid shipping three independent semantics without a matrix doc. If multiple syntaxes parse, define migration and make conflicts loud.

10. Edge cases and failure modes

  • Composition: list + expression both set → error (recommended) or deterministic winner (must be documented).
  • Exclude everything: polling should still advance refs; no infinite queue thrash.
  • Parser attacks: even "mini" languages need max input length and clear token errors.
  • Rollback: revert YAML → revert behavior, no migrations.

11. Open questions

  • Stay on path.Match or adopt a slightly richer wildcard (still non-regex) for branch segments?
  • Do we want one blessed syntax, or branches_exclude as a compatibility shim forever?
  • If we add regex, what are the hard limits (bytes, timeouts, disallowed features)?
  • Should skip logs print the evaluated mode (exclude:main, expr:…) for grepability?
